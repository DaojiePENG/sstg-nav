移植说明 — 供新工程师对接 SSTG 到新机器人平台

本页用于说明 SSTG 在新机器人上的移植边界、需要适配的文件、推荐迁移步骤和验收方法。 建议先以当前 Yahboom X3 版本作为参考 profile，再将平台相关部分替换为目标机器人的实现。

1. 移植目标与适用范围
适用对象
支持 ROS2 + Nav2 的移动机器人
目标平台应至少具备底盘控制、定位、激光雷达和导航执行能力。 如果同时具备 RGB-D 相机，可完整复用搜索、场景理解和 UI 相机链路。
复用范围
上层算法与交互链路通常可直接复用
一般无需修改 interaction_manager、nlp_interface、 navigation_planner、navigation_executor、 map_manager 和前端聊天流程。
当前已验证参考平台为 Yahboom X3 + RPLidar + Orbbec Gemini，可作为迁移基线。
迁移工作的重点是平台适配层，而不是上层算法层。
2. 系统边界
算法层
SSTG Core
负责任务编排、意图识别、拓扑规划、导航执行、语义地图和前端消息桥接。 这一层以标准 ROS2 topic/service/action 为接口，迁移时通常不改业务逻辑。
适配层
Robot Platform Adapter
负责目标机器人的驱动启动、TF/URDF、定位融合、Nav2 参数、视觉入口和 topic 归一。 迁移工作主要集中在这一层。
平台适配层需要向上提供稳定的 /cmd_vel、/odom、/scan、/navigate_to_pose 和标准 TF 树。
如需完整复用搜索与场景理解，还需要提供 /camera/color/image_raw、/camera/depth/image_raw 和 camera_info。
3. 需要适配的文件
文件 职责 迁移时关注点
sstg_rrt_explorer/launch/library/hardware_bringup.launch.py 启动底盘、雷达、URDF、静态 TF 等平台驱动 替换为目标平台的驱动与 bringup 逻辑。
sstg_rrt_explorer/param/common/ekf_x3_override.yaml 定位融合配置 对齐目标平台的里程计源、IMU 话题和 frame 定义。
sstg_rrt_explorer/param/common/nav2_params.yaml Nav2 参数 profile 校准机器人尺寸、运动模型、速度阈值、雷达 topic 和 costmap 参数。
sstg_perception/launch/perception.launch.py
sstg_perception/config/perception_config.yaml		 视觉输入入口 对齐 RGB、Depth、CameraInfo 对应的话题名称与存储路径。
sstg_system_manager/.../webrtc_camera_bridge.py UI 相机桥接 确保前端实时相机流与目标平台视觉话题保持一致。
建议将目标平台逐步整理为独立的 robot profile。这样后续新增机器人时，可以复用同一套移植结构，而不是在业务代码中分散修改。
4. 推荐移植流程
1 确认目标机器人接口
先梳理目标平台可提供的底盘控制、定位方式、雷达、相机、TF 树和 Nav2 action。 推荐先列出 topic / service / action / frame 对照表。
2 替换平台 bringup
修改 hardware_bringup.launch.py，接入目标底盘、雷达、URDF 和静态 TF。 这一阶段的目标是让底层驱动和基础 TF 正常启动。
3 配置定位与 Nav2
根据目标平台的里程计和 IMU 话题重配 ekf_x3_override.yaml，并根据机体尺寸和运动学重配 nav2_params.yaml。
4 配置视觉输入
修改 perception 和 WebRTC bridge 的相机话题配置，确保后端感知链路和前端实时预览读到同一套数据源。
5 完成 topic / frame 归一
如果目标平台自带 namespace 或特殊 frame 命名，建议通过 remap、relay 或 launch 参数统一到 SSTG 期望的接口形式。
6 执行完整链路验证
依次验证导航、搜索、场景理解、地图刷新和前端实时画面，确认平台适配层与上层算法层已完整打通。
5. 验收清单
导航接口
ros2 action list | grep navigate_to_pose 可看到导航 action， 且发起导航任务后 /navigation_feedback 有持续反馈。
TF 树
map -> odom -> base_link/base_footprint 稳定存在， 雷达和相机的安装位姿能够正确解析。
传感器话题
/scan、/odom、/camera/* 话题稳定发布， perception 和 UI 都能正常消费。
典型任务
至少完成一次“去客厅”和一次“帮我找书包”，确认导航、搜索和消息桥接三条链路正常。
如果目标平台具备 RGB-D，相机链路、场景理解和搜索流程应全部可用。
如果目标平台只有部分视觉能力，需在验收文档中明确哪些功能可用、哪些功能降级。
